# Plan: Create Adapters to Prevent Tight Coupling Cross Module

## Status: IMPLEMENTATION COMPLETE

The following changes have been made to decouple modules using the Ports & Adapters pattern:

---

## Changes Made

### 1. Shared Kernel Ports (NEW)

Created in `src/shared/kernel/domain/ports/`:

| Port File | Purpose | Status |
|-----------|---------|--------|
| `gl-posting-queue.port.ts` | GL posting queue operations | ✅ Created |
| `user-provisioning.port.ts` | User creation/linkage (HR→User) | ✅ Created |

Updated `src/shared/kernel/index.ts` to export all new ports.

### 2. Finance Module Adapter (NEW)

Created `src/modules/finance/application/adapters/gl-posting-queue.adapter.ts`:
- Implements `GlPostingQueuePort` interface
- Wraps `GlPostingQueueService` for cross-module access
- Exports `GL_POSTING_QUEUE_PORT` symbol

### 3. Asset Module Refactored

- Created `src/modules/asset/application/adapters/asset-finance.adapter.ts`
- Updated `AssetService` to use `AssetFinanceAdapter` instead of direct `GL_POSTING_QUEUE_SERVICE` injection
- Removed `FinanceModule` import from `AssetModule`

### 4. HR Module Refactored

- Created `src/modules/hr/application/adapters/hr-user-provisioning.adapter.ts`
- Created `src/modules/hr/application/adapters/hr-finance.adapter.ts` (updated)
- Updated `EmployeeService` to use `UserProvisioningPort` instead of direct `UserService` injection
- `HrModule` now exports `USER_PROVISIONING_PORT`

### 5. Purchasing Module Refactored

- Updated `ProcurementFinanceAdapter` to use `GL_POSTING_QUEUE_PORT` instead of direct service injection
- Updated `PurchaseOrderService` to use `GL_POSTING_QUEUE_PORT` instead of direct service injection

### 6. Sales Module Refactored

- Updated `SalesFinanceAdapter` to use `GL_POSTING_QUEUE_PORT` instead of direct service injection

### 7. Warehouse Module Refactored

- Updated `GoodsReceiptService` to use `GL_POSTING_QUEUE_PORT` instead of direct service injection

---

## Remaining Tasks (NOT REQUIRED FOR THIS IMPLEMENTATION)

### SelfServiceModule Infrastructure Imports
The `SelfServiceModule` still imports entities from HR and Finance modules directly. This is a larger refactoring that would require:
- Creating HR attendance/employee query ports
- Creating overtime payment service adapters

### AuthModule User Imports
The `AuthModule` imports `UserModule` directly for authentication. This is intentional - authentication is a core cross-cutting concern.

### UserRoleAssignerAdapter
Still imports `RoleTypeOrmEntity` from IAM module. This adapter is in the User module and owns the user-role relationship logic.

---

## Architecture Benefits Achieved

- ✅ **Reduced circular dependencies** - Modules now depend on shared kernel ports instead of concrete implementations
- ✅ **Clear separation of concerns** - Each module owns its data and business logic
- ✅ **Single point of cross-module access** - All cross-module communication flows through defined adapters
- ✅ **Clean architecture compliance** - Outer layers depend on inner layers through ports only
